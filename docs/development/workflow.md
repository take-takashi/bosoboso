# Development Workflow

このドキュメントには、日常の開発フロー、レビュー前の確認、ブランチ運用を記録する。

テンプレートから新しいプロジェクトを作成するときは、チームの運用に合わせてこのファイルを更新する。

## 基本フロー

通常の開発では、TAKTの標準ワークフローを次の入口から利用する。

1. `mise run takt`を実行し、AIとの対話で要件を整理する。
2. `/go`を入力し、提案されたタスクで「Queue as task」を選ぶ。
3. `mise run takt:run`を実行し、キューにあるタスクを隔離worktreeで処理する。
4. `mise run takt:list`を実行し、差分と検証結果を確認して、タスクブランチを管理する。

TAKTの「Execute now」は現在の作業ツリーを直接変更するため、このリポジトリでは使用しない。TAKTを利用できない作業では、次の手動フローを使用する。

1. 作業内容に関連する `docs/` のドキュメントを確認する。
2. 作業用ブランチのworktreeを作成し、その作業ツリーへ移動する。
3. 必要に応じて、仕様、設計、実装の差分を明示する。
4. 小さく変更し、関連する検証を実行する。
5. 変更に合わせてドキュメントを更新する。
6. レビュー前に差分と検証結果を確認する。

TAKTのプロジェクト設定は`.takt/config.yaml`で管理し、`codex`プロバイダーと日本語を使用する。タスク定義、実行ログ、イベントはローカル状態としてGit管理へ含めない。共有する独自ワークフローやfacetを追加する場合だけ、`.takt/workflows/`または`.takt/facets/`をコミットする。

## worktree運用

ファイルを変更する作業では、メイン作業ツリーを直接使用せず、作業用ブランチごとに`git worktree`でリポジトリルートの`.worktree/<作業名>`へ作業ツリーを作成する。編集、検証、コミットは作成した作業ツリー内で行う。

```shell
git worktree add .worktree/<worktree-name> -b <branch-name>
cd .worktree/<worktree-name>
```

`.worktree/`の内容は`.gitignore`で除外し、配置場所を維持する`.gitkeep`だけをGit管理する。作業完了後は、対象ブランチが不要であることを確認してからworktreeを削除する。

`mise run setup`によって有効になる`.githooks/pre-commit`は、メイン作業ツリーまたは`main`ブランチからのコミットを拒否する。フックを`--no-verify`で回避しない。例外的にメイン作業ツリーでの変更が必要な場合は、作業前に理由を明示して合意を得る。

## ブランチ運用

| 項目 | 方針 |
| --- | --- |
| デフォルトブランチ | `main` |
| 作業場所 | `.worktree/<作業名>`に作成した作業ツリー |
| 作業ブランチ名 | 変更内容が分かる短い名前を使う。詳細規則は未決定 |
| マージ方法 | 未決定。リポジトリの保護ルールを確定した時点で更新する |

## レビュー前確認

- 変更内容がテンプレートとして再利用しやすいかを確認する。
- 関連する `docs/` が更新されているかを確認する。
- 実行した検証コマンドと結果を記録する。
- 秘密情報や不要なローカル設定が含まれていないかを確認する。
- 音声ファイル、Whisperモデル、個人の設定・辞書が含まれていないかを確認する。
- 音声または文字起こし内容がログ、テスト出力、PR本文へ含まれていないかを確認する。
- GitHub Actionsの外部Actionは、バージョンコメントを添えて完全なコミットSHAに固定する。

## PR作成前の検証

1. モデル不要の`mise`検証タスクを実行する。
2. リポジトリ外の音声ファイルとWhisperモデルを指定し、Integration Testを実行する。
3. 同じく外部ファイルを指定し、パッケージ済みアプリのE2Eテストを実行する。
4. 実マイクとmacOS固有機能を手動チェックする。
5. PRテンプレートへ、実行したmiseタスク、モデル名、macOSバージョン、Apple Silicon機種、結果を記録する。

モデルを使うテストはCIでは実行しない。音声内容と文字起こし結果そのものはPRへ記載しない。
