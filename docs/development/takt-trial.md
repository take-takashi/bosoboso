# TAKTワークフローの試行

## 目的

`bosoboso-tdd`ワークフローを使い、仕様策定、人間承認、TDD実装、独立レビュー、受け入れまでを小さな機能で一巡させる。

最初の題材には、設定ファイルの読み込み機能を使用する。期待挙動は`docs/specs/configuration.md`を正として、TAKTの仕様策定ステップで既存仕様の不足や矛盾を洗い出す。

## 試行するタスク

`mise run takt`を実行し、`bosoboso-tdd`ワークフローを選択して、次のタスクを入力する。

```text
docs/specs/configuration.mdに従い、bosobosoの設定ファイル読み込み機能を実装してください。

今回の対象は、設定ファイルの読み込み、TOMLの解析、必須項目の検証、利用者が修正可能なエラーの返却までです。
設定ファイルの探索、GUI表示、Whisperモデルのロード、辞書ファイル本体の読み込みは対象外です。

仕様に不足や矛盾がある場合は実装を開始せず、仕様案、テスト観点、実装計画に明示して人間の承認を求めてください。
Go実装はt-wada式TDDで進め、テストリスト、RED、GREEN、REFACTORの結果を記録してください。
```

対話で要件が整理されたら`/go`を入力し、「Queue as task」を選択する。「Execute now」は選択しない。

## 実行と承認

キューへ登録したタスクは次のコマンドで実行する。

```shell
mise run takt:run
```

ワークフローは次の二箇所で停止し、人間の入力を待つ。

1. 仕様案、テスト観点、実装計画を確認する仕様承認
2. 独立レビューと受け入れ結果を確認する最終承認

仕様承認では、少なくとも対象範囲、対象外、エラーの分類、境界値、受け入れ条件を確認する。最終承認では、受け入れ条件別の証拠、未実施の手動確認、残存リスク、APIキーや外部有料サービスが追加されていないことを確認する。

## 結果の確認

実行後は次のコマンドでタスクブランチと結果を確認する。

```shell
mise run takt:list
```

初回試行では、ワークフロー自体の評価も行う。

- 承認前に実装が開始されなかったか。
- TDDのRED、GREEN、REFACTORを追跡できたか。
- 実装者とは独立した観点でレビューと受け入れが行われたか。
- 人間へ提示された情報が承認判断に十分だったか。
- 同じ指摘による修正ループが過剰に続かなかったか。

改善点が見つかった場合は、個別タスクの都合ではなく、今後も再利用するルールだけを`.takt/workflows/bosoboso-tdd.yaml`または`docs/lessons.md`へ反映する。
